This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A two-stage pipeline that pulls FRC team data from The Blue Alliance (TBA) API and mines it for sponsor information:

1. `export_teams.py` — fetches all teams active in the 2025 season from TBA and writes `teams.csv`.
2. `extract_sponsors.py` — reads `teams.csv` and writes `sponsors.json`, a deduplicated sponsor list.

Both `teams.csv` and `sponsors.json` are generated output, gitignored, and regenerated by running the scripts — never hand-edit them.

## Commands

```sh
# Setup (Windows; venv already created with Python 3.14 via scoop)
.venv\Scripts\activate
pip install -r requirements.txt

# One-time: get a key from https://www.thebluealliance.com/account
cp .env.example .env
# edit .env and set TBA_AUTH_KEY=<your key>

# Run the pipeline (order matters — extract_sponsors.py requires teams.csv to exist)
python export_teams.py
python extract_sponsors.py
```

There is no test suite, linter, or build step configured for this project.

### Querying sponsors.json

```sh
# Top N sponsors overall, by team count
jq --argjson n 10 'sort_by(-.qty)[:$n] | map({sponsor, qty})' sponsors.json

# Top N sponsors in Utah, by Utah team count
jq --argjson n 10 '[.[] | select(.qty_utah > 0)] | sort_by(-.qty_utah)[:$n] | map({sponsor, qty_utah})' sponsors.json
```

## Architecture

**TBA client quirk:** the package `tba-api-client-python` is installed straight from GitHub (not on PyPI). Its pip distribution name (`tba-api-v3client`, used in `requirements.txt`) does not match its import name (`tbaapiv3client`, used in code) — this mismatch is easy to reintroduce if the dependency is ever touched.

**`export_teams.py`:**
- Auth key loaded from `TBA_AUTH_KEY` env var via `python-dotenv` / `.env` (never committed — `.env.example` is the template).
- Pages through TBA's `TeamApi.get_teams_by_year(year, page_num)` (500 teams/page) until an empty page is returned. Uses the *full* `Team` model rather than `TeamSimple`, since `TeamSimple` lacks the `website` field.
- `YEAR` is hardcoded to 2025 (defines what "active" means — teams that competed that season). Update this constant to shift the target season.

**`extract_sponsors.py`** — the core logic worth understanding before touching it:
- TBA has no dedicated sponsor field. The `name` field is unstructured text shaped like `"Sponsor1/Sponsor2/Sponsor3&School Name"`, and it's messy: sponsor names can themselves contain `&` (e.g. "Florida Power & Light"), and some teams chain multiple school/org affiliations with extra `&`s.
- Parsing heuristic (`parse_sponsors`): if `name` has no `&` at all, the team has no listed sponsors (it's a bare school name). Otherwise split on `/`; every segment except the last is a sponsor as-is. The last segment is split on its *rightmost* `&` only — the part before is the final sponsor, the part after is discarded as the school name.
- This is deliberately a best-effort heuristic, not exact — roughly 800 of 3731 teams have more than one `&` in their name and can produce a stray fragment in the sponsor or school position. This is a known, accepted limitation of TBA's raw data, not a bug to chase.
- Sponsor name matching for dedup: trim + collapse internal whitespace, then exact case-sensitive comparison (no case-folding — avoids merging distinct orgs that happen to share a name in different cases).
- Output records: `{sponsor, qty, qty_utah, teams: [{team_number, state}, ...]}`, sorted by `qty` descending then sponsor name ascending. `qty_utah` counts how many of that sponsor's teams have `state == "Utah"`.

**`csv_utils.py`** — shared `csv_safe()` guards against CSV formula injection (values starting with `=`, `+`, `-`, `@`, tab, or CR get a leading `'` prefix) since team names originate from third-party TBA data. Only used by `export_teams.py` — `extract_sponsors.py` writes JSON, where this isn't a risk (JSON string escaping via `json.dump` is inherently safe here).
